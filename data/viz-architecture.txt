# Ledger of Meluhha — Visualization Architecture
## Third Buyer Advisory LLC | Venugopal 2026


---


## What It Is


A single self-contained HTML file. User drags a database file
onto the page. Everything renders live — geographic trade map,
decoded seals, sign frequency, weight codebook — all from live
queries against the Alloy-verified corpus schema. No server.
No build chain. No framework. Just a browser and a database file.

## Design Language


### Visual identity
- Header link: thirdbuyeradvisory.org
- Background: #0f0c07 (near-black, warm)
- Accent: #b5121b (TBAred)
- Primary text: #f0e6d0 (warm sand)
- Data color: #c4924a (clay)
- Display font: Cormorant Garamond — narrative, conclusions,
  titles. Weight 300 for large headings. Italic for emphasis.
- Data font: JetBrains Mono — sign numbers, field labels,
  weights, coordinates, methodology notes.
- Font is epistemological status. Cormorant = interpretation.
  JetBrains Mono = fact.


### Tufte rules — non-negotiable
1. Data-ink ratio. No chartjunk. No grid boxes. No panel borders
   around data. No background fills on charts. Every mark earns
   its place.
2. Direct annotation. Routes labeled at midpoint. Sites labeled
   at the dot. No floating legend box anywhere on the page.
3. Small multiples. All routes visible simultaneously at low
   opacity. Hover brings one forward. The comparison between
   iron moving north and jar goods moving west is the argument —
   it must be visible simultaneously, not sequentially.
4. One claim sentence per panel. Above every visualization, a
   declarative sentence stating what the data shows. The chart
   proves the sentence. Not a title. A conclusion.
5. The detail is the hero. The decoded seal field breakdown —
   F1 through F5, sign number, concordance path, resolved value —
   is shown large and complete, not tucked away.

## Width Encoding — Sankey Routes on Geography


Route stroke-width is proportional to relative trade volume
derived from cuneiform Meluhha records. This is the primary
analytical contribution of the visualization.


| Route                         | Width | Evidence basis |
|-------------------------------|-------|----------------|
| Jar goods to Mesopotamia      | 8px   | S-342 = 10% of entire corpus |
| Textiles to Mesopotamia       | 6px   | Primary bulk cuneiform record |
| Copper-bronze to Magan/Dilmun | 5px   | Khetri mines to Lothal to Oman |
| Carnelian to Dilmun           | 4px   | Luxury export, lower volume |
| Iron: Tamil Nadu to IVC       | 3px   | Fish sign supply corridor |
| Timber and Ivory              | 2px   | Sporadic high-value lots |


Width calibration bar drawn directly on map, bottom-left.
Annotated in place: "Width proportional to relative trade volume
(cuneiform records)." No legend box. One reference mark.


### The crossing


The iron supply corridor — green, thin, running south-to-north
from Tamil Nadu — crosses the jar goods maritime route — clay,
thick, running west toward Mesopotamia — near the Gujarat coast.
Lothal sits exactly at the crossing node.


This crossing is the paper's argument made visible. Iron moving
up. Bulk goods moving out. Two ends of one ledger. Width encoding
makes the asymmetry immediate — thick westbound export, thin
northbound supply. The eye resolves the argument before the
text is read.

relative volume, cuneiform source.

supply zone, Mesopotamian destination, resource zone).

all others dim to 10%. Click again to reset.


Click commodity manifest row: filter map to routes
carrying that commodity only.

## Data Flow


Database file dropped onto page. FileReader reads binary into
ArrayBuffer. sql.js WASM opens it in memory. Queries execute
against live schema. D3 renders all panels.


No intermediate files. No server round-trips. No JSON export.
No build step. The HTML file is the artifact.


---


## Queries to Panels


Trade route map: sites table for coordinates and type,
routes hardcoded with width values from volume table above.


Sign frequency: base_sign table, source MAHADEVAN,
ordered by total descending, cumulative running sum.


Decoded seals: cisi_inscription joined to sign_concordance,
filtered to M-52A and M-148A, joined to codebook for
field role resolution.


Weight codebook: hardcoded from base unit 0.856g.
Binary series times 1/2/4/8/16/32/64.
Decimal series times 160/200/500/1000.


Rajan-Sivanantham composite chart: base_sign table,
source RS2025, ordered by composites descending, top 10.
Sign 2.0 at 179 composites and Sign 20.0 at 0 composites
annotated directly on the chart.


Tamil Nadu parallels: morphological_parallel table,
sign_a_src RS2025 joined to MAHADEVAN and CISI.
Confidence level drives opacity on map.

## Constraints — Single HTML File


The entire application is one HTML file. No CDN calls at runtime.
No external dependencies. No internet connection required after
the file is saved.


Plotly.js compiled in. The Plotly bundle is embedded directly
inside the HTML file as an inline script. One file, opens
anywhere, works offline. Size will be approximately 3–4MB —
acceptable for a research artifact.


sql.js WASM compiled in. The WebAssembly binary for SQLite is
base64-encoded and embedded inline. No CDN. No locateFile path.
The WASM module initialises from the embedded binary on load.


D3.js compiled in. The D3 bundle embedded inline alongside
Plotly for the geographic map rendering.


Result: the HTML file is fully self-contained. Drop it on a
USB stick, open it on an air-gapped machine, drag the database
file onto it — everything works. That is the correct constraint
for an archival research artifact.


The database files ship separately in the Zenodo zip.
The HTML file needs no network access whatsoever.


### Chart library division


D3 handles the geographic map — projection, curved route paths,
site dots, direct annotation, width-encoded Sankey strokes.
D3 is the right tool for geographic SVG.


Plotly handles the analytic charts — sign frequency cumulative
curve, weight codebook horizontal bars, Rajan-Sivanantham
composite bar chart. Plotly produces cleaner interactive charts
with less code for these panel visualizations.


Both libraries embedded inline. No conflict.

## Excel and SQLite — Direct Connection


No Excel export script. No intermediate files. Excel connects
directly to the SQLite databases.


Data → Get Data → From Database → From SQLite Database.
Point at indus_corpus.db or indus_codebook.db.
Every table loads as a named Power Query.
Joins, filters, and pivots work on the live data.
Refresh pulls from the database file directly.


This means researchers get full analytical access without
any export step. The database is the source of truth.
Excel is the analysis surface. They stay in sync.


Platform note: on Windows, SQLite ODBC is straightforward.
On Mac, the sqliteodbc driver from Christian Werner is required
before the Excel connection works. Worth installing before
sitting down with the data.


The Zenodo zip ships the two database files. Researchers
connect Excel themselves. No workbook to maintain.
No export script to run.
